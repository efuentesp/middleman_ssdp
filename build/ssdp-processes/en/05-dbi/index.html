Database Design and Implementation

The process of transforming the logical model data structure model into a physical database model capable of satisfying data access requests from functional components, to store expected data volumes, to satisfy performance and concurrency requirements, and to preserve stored data integrity and privacy.

Process Diagram


Process Description

Inputs Steps Outputs
  • DOM. Logical Entity-Relationship model / Domain Model (updated)
  • NFR. Non-Functional Requirements Specification(estimated volume and performance)
  • DBS. Database Standards
  • ARS. Application's Architecture Specification
  • SEC. SSecPro Security Standards
  • FCS. Functional Component specifications [optional, to supplement the inspection and for access optimization]
  1. Inspect data structure model - supports persistence and data access requests described in functional component specifications.
  2. Identify physical database implementation standards, guidelines and constraints (provided by client).
  3. Develop logical database model and define integrity rules and constraints (primary keys, foreign keys, domain constraints).
  4. Normalize logical database model (3NF) and include redundant data-structures (de-normalization and consolidated repositories) for performance optimization.
  5. Identify control columns for concurrency management and logging mechanisms (timestamps, record versions, creation and update users, etc.).
  6. Design physical data structures (tables, columns, column attributes, etc.) complying with naming conventions and standards.
  7. Design database instance landscape (instances for development, testing and production; required locations, spaces and configuration).
  8. Implement database instances (at least development and testing)
  9. Design and implement database access security (considering security requirements, architecture, Configuration Management Document and SsecPro Safety Standards).
  10. Identify and implement database objects (indexes, views, constraints and triggers).
  11. Verify physical database model against volume and performance requirements.
  12. Establish physical database model baseline and change management process.
  13. Specify domain data structure model mapping to physical database model (ORM).
  14. Develop and test application data access layer based on ORM (Object Relational Mapping).
  15. Design and implement archiving and backup/restore procedures [optional].
  16. Post-mortem DBI.
  • PDM. Physical Database Model (DDL).
  • DBI. Development and Testing Database Instances
  • ORM. Object Relational Mapping (domain data structure model to physical database model mapping)
  • DAL. Application Data-Access Layer (DB-access functions or Object-relational mapping components) [implemented and tested]

Roles

Name Description
PL. Project Leader Plan activities.
DBA. Database Administrator Lead the database design and implementation activities and produce deliverables.
BA. Business Analyst Provide domain data structure model, functional component specifications, and volume estimates and performance requirements.
SA. Software Architect Implements data access layers based on ORM (Object Relational Mapping).
CDA. Client DBA Provide standards and constraints, and validate deliverables.
