Architecture Design and Implementation

ARC consists in making technical decisions on the application to be implemented and the platform on which will be executed, besides the application’s framework to be implemented and the support subsystems that meet the functional and non-functional requirements set.

Process Diagram


Process Description

Inputs Steps Outputs
  • NFR. Non-Functional Requirements Specification
  • COM. Application Common Features Specification
  • FRS. Functional Requirements Specification
  • SFMK. Softtek's Application Framework, support subsystems and reusable components (from Softtek’s repository, if exist)
  • CFMK. Client's Framework application, support subsystems and reusable components (from client’s repository, if exist)
  • FPI. Functional Patterns Inventory and their specifications (from Softtek’s repository, if exist)
  • SEC. SSecPro Security Standards
  • IPI. Implementation Patterns Inventory (from Softtek’s repository, if exist)
  1. Inspect non-functional requirements specifications and common features (include client representatives infrastructure and architecture).
  2. Select representative use cases for the application architecture.
  3. Identify types of functional components (Functional Components Taxonomy).
  4. Specify the platform and infrastructure available for development, test and production environment and application transport considerations for implementation by the various environments.
  5. Select reusable architectural patterns, reference architectures, application frameworks and support subsystems to cover and provide a solution to non-functional requirements as necessary regarding security, scalability, availability, maintainability and performance as well as the common features for each type functional component.
  6. Planning tasks to Architecture design and implementation.
  7. Specify the functional patterns and standards for external interfaces for each type of functional component.
  8. Design software architecture (create software architectural views) and document technical decisions.
  9. Develop/reuse application’s framework.
  10. Develop/reuse support subsystems and reusable software components.
  11. Specify an implementation pattern for each type of functional component.
  12. Specify implementation guidelines (naming conventions, coding and style standards, coding practices, resource usage guidelines, technical restrictions).
  13. Develop a reference implementation for each type of functional component according to functional and implementation patterns.
  14. Test reference implementation against non-functional requirements concerning security, scalability, availability, maintainability and performance.
  15. Inspect the application framework and reference implementation on compliance with security requirements.
  16. Validate architecture specifications, application framework and reference implementations (with customer IT infrastructure and architectural representatives).
  17. Post-mortem ARC.
  • FDG. Functional design guidelines:
    • Functional Components Taxonomy.
    • Functional Pattern Specifications.
    • User Interface Standards.
  • ARS. Application’s Architecture Specification:
    • General Architecture Models (Technology Stack (HW and SW), Physical View (HW and SW), Logical View (layers and part´s type), Deployment View, Integration View (contracts and interlayer sequences) )
    • Specification for implementation’s pattern
  • ARCI. Application Architecture´s Implementation (tested):
    • Application’s Framework.
    • Support Subsystems.
    • Reusable Software Components.
  • IMPG. Implementation Guidelines:
    • Naming conventions
    • Coding Standard, practices and style.
    • Resources Usage Guidelines.
    • Technical Restrictions.
  • FCRI. Reference Implementation for each type of functional component (tested).
  • ARAU. Audit Report of the Architecture Security.
