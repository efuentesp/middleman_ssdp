Desenho e Implementação da Base de Dados

É o processo de transformar o modelo de dados lógico em um modelo físico da base de dados, capaz de satisfazer as demanda de acesso aos dados desde os componentes funcionais, armazenar os volumes de dados esperados, satisfazer os requisitos de desempenho e concorrência, e preservar a integridade da informação armazenada e a privacidade.

Diagrama do Proceso


Descrição do Proceso

Entradas Passos Saídas
  • DOM. Modelo lógico da base de dados.
  • NFR. Requisitos de volumes estimados e de desempenho.
  • FCS. Especificação de componentes funcionais [opcional, para complementar a inspeção e para otimização de acessos].
  1. Inspecionar o modelo de estrutura de dados – suporta a persistência e solicitações de acesso aos dados descritos nas especificações de componentes funcionais;
  2. Identificar os padrões, diretrizes e restrições para a implementação física da base de dados (fornecido pelo cliente);
  3. Gerar o modelo lógico da base de dados e definir regras de integridade e restrições (chaves primárias, chaves estrangeiras e restrições de domínio);
  4. Normalizar o modelo lógico da base de dados (3º Forma Normal) e incluir estruturas de dados redundantes (desnormalização e consolidação de repositórios) para otimização de desempenho;
  5. Identificar colunas de controle para administração de concorrência e mecanismos de bitácora (registro de tempo, registro de versões, usuários que foram criados e atualizados);
  6. Desenhar a estrutura da base de dados (tabelas, colunas, atributos das colunas) cumprindo com as convenções de nomenclatura e padrões;
  7. Desenhar o panorama de instâncias da base de dados (instâncias para desenvolvimento, testes e produção; localização requerida, espaços e configuração);
  8. Implementar as instâncias da base de dados (desenvolvimento e testes);
  9. Desenhar e implementar a segurança de acesso a base de dados;
  10. Identificar e implementar objetos da base de dados (índices, views, constraints, triggers);
  11. Verificar o modelo físico da base de dados contra os requisitos de volume e de desempenho;
  12. Estabelecer a Baseline do modelo de dados físico e processo de administração de mudanças;
  13. Especificar o mapeamento entre o modelo de estrutura de dados de domínio e o modelo físico da base de dados (ORM);
  14. Desenvolver e testar a camada de acesso a dados da aplicação baseada no ORM (Mapeamento Relacional de Objeto);
  15. Desenhar e implementar procedimentos de armazenamento e de backup / recuperação [opcional];
  16. Post-mortem DBI.
  • PDM. Modelo físico da base de dados (DDL).
  • DBI. Instâncias da base de dados de desenvolvimento e testes.
  • ORM. Mapeamento Relacional de Objeto (mapeamento da estrutura de dados de domínio ao modelo físico da base de dados).
  • DAL. Camada de acesso a dados da aplicação (funções de acesso a base de dados ou componentes de mapeamento objeto-relacional) [Implementados e testados].

Papéis

Nome Descrição

SA. Arquiteto de software Fornecer volume estimado e requisitos de desempenho.
CDA. DBA do Cliente Fornecer padrões e restrições, assim como validar resultados.
